# memo — Техническая спецификация

> Документ описывает технологический стек, выбор AI-моделей, стратегию оптимизации и целевые характеристики производительности приложения **memo**.

---

## 1. Технологический стек

| Слой | Технология | Назначение |
|---|---|---|
| Десктоп-оболочка | Tauri 2 | Кроссплатформенная упаковка, нативный доступ к ФС, лёгкий системный WebView вместо встроенного Chromium |
| UI | React + TypeScript | Двухпанельный интерфейс |
| Backend | Python 3.11+ / FastAPI | Бизнес-логика, REST API на localhost |
| Запуск backend | Tauri sidecar | Python-процесс стартует и завершается вместе с приложением |
| Оркестрация RAG | LangChain | Парсинг, чанкинг, retrieval-конвейер |
| Векторная БД | ChromaDB (persistent) | Локальное хранилище эмбеддингов, HNSW-индекс |
| Реляционная БД | SQLite (SQLAlchemy) | История чатов, состояние индекса |
| LLM-движок | Ollama | Локальный запуск всех моделей |
| Парсинг PDF | PyMuPDF (fitz) | Быстрое извлечение текста |
| Парсинг DOCX | python-docx | Извлечение текста из Word |
| Мониторинг ФС | watchdog | Отслеживание изменений файлов |

**Принцип связи слоёв:** React-панели общаются между собой через Tauri IPC; UI обращается к Python-backend по HTTP на `localhost` (порт назначается при старте); backend — единственный, кто обращается к Ollama, ChromaDB, SQLite и файловой системе.

## 2. Выбор AI-моделей

### 2.1 Контекст выбора

Ключевое требование — работа на слабых устройствах **без дискретной видеокарты** (CPU-инференс). По состоянию на середину 2026 года практический потолок для CPU-инференса — модели 3–4B параметров в квантизации Q4; они дают 5–15 токенов/сек на современных 8-ядерных процессорах при 8–16 ГБ RAM. Модели крупнее 7–8B на CPU работают слишком медленно для интерактивного использования.

Поскольку целевые документы преимущественно на русском языке, приоритет отдаётся **мультиязычным** моделям (Qwen, Gemma), а не англоцентричным (Phi, базовый Llama).

### 2.2 Модели по функциям

| Функция | Модель | Размер (Q4) | Кто выбирает |
|---|---|---|---|
| Чат / Q&A | qwen3:4b (рекомендуемая) | ~2.5 ГБ | Пользователь, из установленных в Ollama |
| Создание документов | та же, что для чата | — | Пользователь |
| Эмбеддинги (индексация, RAG, организация) | bge-m3 | ~1.2 ГБ | Предустановлена |
| Именование папок (авто-организация) | qwen3:1.7b | ~1.0 ГБ | Предустановлена |

### 2.3 Рекомендуемые модели для чата (CPU)

Пользователь выбирает любую установленную модель, но в документации и onboarding рекомендуются проверенные для CPU варианты:

| Модель | Параметры | Сильные стороны | Скорость на CPU |
|---|---|---|---|
| qwen3:4b | 4B | Мультиязычность (вкл. русский), баланс качества | ~8–12 ток/с |
| qwen2.5:3b | 3B | Легче, быстрее | ~10–15 ток/с |
| gemma3:4b | 4B | Хорошая мультиязычность | ~8–12 ток/с |
| llama3.2:3b | 3B | Быстрая, но англоцентричная | ~10 ток/с |
| phi4-mini | 3.8B | Капризна к русскому, сильна в рассуждениях | ~12 ток/с |

**Рекомендация по умолчанию:** `qwen3:4b` — лучший баланс качества и мультиязычности для русскоязычных документов на CPU.

### 2.4 Эмбеддинг-модель: bge-m3

Выбрана `bge-m3` (BAAI), а не популярная `nomic-embed-text`, по причинам:

- **Мультиязычность.** bge-m3 поддерживает 100+ языков, включая русский. nomic-embed-text v1 англоцентрична и даёт низкое качество retrieval на русском.
- **Не требует префиксов.** В отличие от моделей Nomic/Qwen, bge-m3 не требует приставок `search_query:` / `search_document:`, что упрощает конвейер.
- **Приемлемая стоимость на CPU.** Эмбеддинг — разовая операция на документ (результат кэшируется), поэтому 568M параметров не создают проблемы при интерактивной работе.

**Лёгкая альтернатива для очень слабых машин:** `multilingual-e5-small` — меньше и быстрее, ценой некоторого качества retrieval. Может предлагаться как опция в настройках.

### 2.5 Модель для именования папок: qwen3:1.7b

Авто-организация (см. спецификацию компонентов) основной объём работы выполняет через эмбеддинги + кластеризацию, а генеративная модель нужна только для присвоения имён небольшому числу кластеров (обычно 3–8 вызовов). Поэтому здесь достаточно компактной `qwen3:1.7b` — она предустановлена и не настраивается пользователем.

**Оптимизация:** при недоступности или для ускорения предусмотрен fallback без LLM — имя папки формируется из топ-ключевых слов кластера (TF-IDF). Качество ниже, но работает мгновенно и без модели.

## 3. Стратегия оптимизации под слабые устройства

Оптимизация — сквозное требование. Применяются следующие приёмы.

### 3.1 Уровень моделей
- Дефолт — модели 3–4B в квантизации **Q4_K_M**.
- Тюнинг числа потоков Ollama под количество ядер CPU (`OLLAMA_NUM_THREADS`).
- Управление выгрузкой простаивающих моделей (`keep_alive`), чтобы не держать RAM занятой.

### 3.2 Переиспользование эмбеддингов
Документ эмбеддится **один раз**. Полученные векторы используются и для RAG-поиска, и для авто-организации (кластеризация на тех же векторах). Повторного эмбеддинга при организации не происходит.

### 3.3 Инкрементальная индексация
- Для каждого файла хранится `file_hash` и `mtime`.
- При запросе индексации файлы с неизменившимся хэшем **пропускаются**.
- Полная переиндексация папки не требуется — обрабатываются только новые и изменённые файлы.

### 3.4 Ограничение контекста retrieval
- Из векторной БД извлекается ограниченный top-k (по умолчанию k = 4–6 чанков).
- Меньший контекст → меньше токенов на вход LLM → **быстрее генерация на CPU** (узкое место CPU — длина контекста).

### 3.5 Чанкинг
- Размер чанка ~512 токенов, перекрытие ~64 токена.
- Баланс между качеством retrieval и размером индекса.

### 3.6 Стриминг ответов
Ответы LLM стримятся токен за токеном. На CPU это критично: пользователь видит первые слова через ~1–2 сек, а не ждёт весь ответ.

### 3.7 Асинхронность backend
FastAPI работает асинхронно. Долгие операции (индексация, организация) не блокируют UI — он остаётся отзывчивым, прогресс отображается в реальном времени.

### 3.8 Кэширование
- Кэш извлечённого текста документов (чтобы не парсить PDF повторно).
- Persistent ChromaDB — индекс сохраняется между сессиями, не строится заново при запуске.

### 3.9 Батчинг эмбеддингов
Чанки эмбеддятся пакетами, а не по одному — снижает накладные расходы на вызовы модели.

## 4. Целевые характеристики производительности

### 4.1 Аппаратный baseline

| Уровень | Конфигурация | Ожидаемое поведение |
|---|---|---|
| Минимум | 4 ядра, 8 ГБ RAM, без GPU, AVX2 | Работает на моделях 2–3B, чат ~5–10 ток/с |
| Рекомендуемый | 8 ядер, 16 ГБ RAM, без GPU | Модели 4B комфортно, чат ~10–15 ток/с |
| Комфортный | 8+ ядер, 16+ ГБ RAM, iGPU (Radeon 780M+/Intel Arc) | Ollama офлоудит слои на iGPU, ~15–25 ток/с |

### 4.2 Ориентиры по операциям (на рекомендуемом baseline)
- Первый токен ответа: 1–3 сек.
- Индексация: десятки документов в минуту (зависит от объёма текста).
- Авто-организация 20–50 файлов: единицы минут (эмбеддинги + кластеризация + несколько вызовов LLM для имён).

## 5. Зависимости и окружение

### 5.1 Системные требования
- **Ollama** установлена и запущена (внешняя зависимость, не входит в дистрибутив).
- ОС: Windows / macOS / Linux (кроссплатформенность через Tauri).

### 5.2 Предустановленные модели
При первом запуске приложение проверяет наличие и при необходимости предлагает скачать (`ollama pull`):
- `bge-m3` — эмбеддинги (обязательно)
- `qwen3:1.7b` — именование папок (обязательно)
- `qwen3:4b` — рекомендуемая модель чата (опционально, пользователь может выбрать другую)

### 5.3 Ключевые Python-зависимости
fastapi, uvicorn, langchain, chromadb, ollama (python-клиент), pymupdf, python-docx, sqlalchemy, watchdog, scikit-learn (кластеризация для авто-организации).

### 5.4 Ключевые frontend-зависимости
@tauri-apps/api, react, typescript, библиотека для рендеринга Markdown (для превью .md), библиотека дерева файлов / drag-and-drop.

---

## 6. Принципы, заложенные в систему

- **Полная локальность.** Ни один компонент не обращается в интернет (кроме `ollama pull` при установке моделей).
- **Прозрачность источников.** Ответы RAG всегда сопровождаются названиями файлов-источников.
- **Обратимость опасных операций.** Авто-организация применяется только после подтверждения; до этого все изменения — в превью.
- **Ленивая работа.** Модели и индексы загружаются по необходимости, простаивающие — выгружаются.
