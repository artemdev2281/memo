## Итоговые находки (10 наиболее критичных)

### 1. `indexer.py` — `collection.add()` вне try/except

**Файл:** `src/backend/memo/services/indexer.py` ~строка 120  
**Серьёзность:** Критическая

`collection.add()` не обёрнут в try/except. Ошибка там (например, несоответствие длин `ids`/`embeddings`) прерывает async-генератор и закрывает SSE-стрим без записи `IndexState` и без структурированного события ошибки. Файл остаётся в неопределённом состоянии в БД.

**Сценарий:** Ollama возвращает N-1 эмбеддингов для N чанков → `collection.add()` бросает `ValueError` → стрим рвётся → пользователь видит общее сообщение об ошибке без указания файла.

---

### 2. `chroma.py` — `get_collection()` не потокобезопасна

**Файл:** `src/backend/memo/chroma.py` ~строка 12  
**Серьёзность:** Критическая

`if _client is None` без блокировки. Asyncio event loop и watchdog-потоки могут войти в ветку одновременно и создать два `PersistentClient` на одном пути — ChromaDB не поддерживает это.

**Сценарий:** первая индексация + одновременное событие `on_deleted` → два клиента → SQLite exclusive lock → исключение или повреждение векторного хранилища.

---

### 3. `indexer.py` — `except: pass` скрывает ошибки удаления из Chroma

**Файл:** `src/backend/memo/services/indexer.py` ~строка 115  
**Серьёзность:** Высокая

`except Exception: pass` вокруг `collection.get()` / `collection.delete()` скрывает все ошибки. Если `collection.delete()` упал молча, последующий `collection.add()` добавит дубликаты (или бросит `DuplicateIDError`), и файл навсегда выпадет из переиндексации (hash в БД совпадает → файл пропускается).

---

### 4. `watcher.py` — нет обработчика `on_created`

**Файл:** `src/backend/memo/services/watcher.py` ~строка 15  
**Серьёзность:** Высокая

`_Handler` обрабатывает только `on_modified`, `on_moved`, `on_deleted`. Новые файлы, скопированные в отслеживаемую папку, игнорируются: они не попадают в `IndexState` и не будут проиндексированы командой "Обновить устаревшие".

---

### 5. `main.py` — порт печатается до завершения bind() в uvicorn

**Файл:** `src/backend/memo/main.py` ~строка 47  
**Серьёзность:** Высокая

`print(f"MEMO_PORT={port}")` выполняется до `uvicorn.run()`. Rust-поток сразу кеширует порт; frontend может отправить запрос до того, как сервер начал слушать сокет.

**Сценарий:** на медленной машине под Windows frontend получает `ERR_CONNECTION_REFUSED` при старте, нет логики повторных попыток → постоянный баннер ошибки.

---

### 6. `fs.py` — `startswith` для путей даёт ложные совпадения

**Файл:** `src/backend/memo/services/fs.py` ~строка 11  
**Серьёзность:** Средняя

```python
r.file_path.startswith(abs_path)
```
Строковое сравнение без разделителя: `/home/user/doc` совпадает с `/home/user/documents/file.txt`.

**Сценарий:** в дереве для `/home/user/doc` появляются статусы файлов из `/home/user/documents/` — ложные иконки ✅/⚠️ у посторонних файлов.

**Исправление:** `r.file_path.startswith(abs_path.rstrip("/\\") + "/")`

---

### 7. `api/index.ts` — `res.body!` без проверки на null

**Файл:** `src/frontend/src/api/index.ts` ~строка 28  
**Серьёзность:** Средняя

`res.body!.getReader()` без guard. В некоторых средах Tauri WebView `fetch()` может вернуть `Response` с `body === null` → `TypeError`, минующий обработку ошибок в UI.

---

### 8. `LeftPanel.tsx` — неверный `total` в прогресс-баре при старте

**Файл:** `src/frontend/src/components/LeftPanel/LeftPanel.tsx` ~строка 54  
**Серьёзность:** Средняя

`setIndexProgress({ done: 0, total: selectedPaths.size })` вызывается до начала SSE-стрима. `index_files` разворачивает директории на сервере, реальное число файлов может быть намного больше. Прогресс-бар показывает неверный знаменатель до прихода первого SSE-события.

---

### 9. `ipc/backend.ts` — нет дедупликации одновременных вызовов `getBackendPort()`

**Файл:** `src/frontend/src/ipc/backend.ts` ~строка 5  
**Серьёзность:** Низкая

Несколько одновременных вызовов до разрешения `_port` запускают независимые циклы из 40 попыток по 250 мс вместо ожидания общего Promise.

---

### 10. `lib.rs` — `child.kill()` при уничтожении любого окна

**Файл:** `src/frontend/src-tauri/src/lib.rs` ~строка 100  
**Серьёзность:** Низкая (для текущего однооконного приложения)

`on_window_event` убивает Python-процесс при `Destroyed` для любого окна. В многооконной конфигурации закрытие вспомогательного окна уничтожит backend при живом главном окне. `guard.take()` также очищает хранимый handle, поэтому финальное событие не сможет повторить попытку.
