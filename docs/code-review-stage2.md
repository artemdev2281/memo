## Критические (потеря данных / полный отказ)

### 1. Ответ не сохраняется при ошибке Ollama
**Файл:** `src/backend/memo/api/chats.py:147`

Когда `answer_stream` выбрасывает ошибку (таймаут, краш Ollama), генератор делает `return` сразу после `yield error`. Ветка `done`, где вызывается `chat_store.add_message`, никогда не достигается. Частичный ответ и сам факт ошибки не записываются в БД.

**Сценарий:** Ollama падает на середине генерации → пользователь видит ошибку в UI → при перезагрузке диалог показывает вопрос без ответа.

**Исправление:** В ветке `error` внутри `stream()` сохранять в БД то, что успело накопиться в `full_content`, с пометкой или просто пустой строкой — чтобы не оставлять висячий user-turn.

```python
elif event["type"] == "error":
    # сохранить частичный ответ, даже если он пустой
    chat_store.add_message(
        chat_id, "assistant",
        "".join(full_content) or "[Ошибка генерации]",
        [], thinking=None
    )
    yield f"data: {json.dumps(event)}\n\n"
```

### 2. Кеш порта не сбрасывается после рестарта бэкенда
**Файл:** `src/frontend/src/ipc/backend.ts:19`

`_port` кешируется навсегда. Если бэкенд крашнул и Tauri перезапустил его на другом порту, `getBackendPort()` возвращает старый порт по быстрому пути (`if (_port !== null) return _port`). Нет механизма сброса.

**Сценарий:** Бэкенд стартовал на 51234 → упал → поднялся на 51799 → все запросы идут на 51234 → ECONNREFUSED → UI полностью сломан до ручного перезапуска окна.

**Исправление:** В `apiFetch` (или в самом `getBackendPort`) при получении сетевой ошибки сбрасывать `_port = null` чтобы следующий вызов снова запустил опрос.

```typescript
// в api/chat.ts и api/index.ts — обернуть fetch в try/catch
} catch (e) {
  if (e instanceof TypeError && e.message.includes('fetch')) {
    resetBackendPort(); // новый экспорт из backend.ts
  }
  throw e;
}

// в backend.ts добавить:
export function resetBackendPort() { _port = null; }
```

## Высокие (неверное поведение, заметное пользователю)

### 3. Переполнение контекстного окна без предупреждения
**Файл:** `src/backend/memo/services/rag.py:150` / `ollama_client.py:94`

`chat_stream` вызывается с `num_ctx=4096` по умолчанию. Сообщения теперь включают до 20 turns истории + RAG-контекст (5 чанков ≈ 1000 токенов) + system prompt ≈ системный + вопрос. Суммарно легко достигается 3500+ токенов, оставляя ~500 на ответ. Ollama молча обрезает ранние сообщения без ошибки.

**Сценарий:** После 6+ обменов с документом модель «забывает» факты, которые уже подтвердила, и противоречит предыдущим ответам.

**Исправление:** Увеличить `num_ctx` до 8192 (или вынести в `settings.py`), и/или передавать его явно из `answer_stream`.

```python
# settings.py — добавить поле
num_ctx: int = 8192

# rag.py — передавать явно
async for chunk in ollama.chat_stream(
    model, messages, num_ctx=settings.num_ctx, think=think
):
```

### 4. Молчаливый переход на параметрическую память при пустом retrieval
**Файл:** `src/backend/memo/services/rag.py:78`

Если пользователь выбрал документы, но retrieval не нашёл релевантных чанков (вопрос не по теме), `context_block` остаётся пустым и `_build_messages` переключается на `_GENERAL_SYSTEM_PROMPT`. Модель отвечает из тренировочных данных без каких-либо оговорок, а `sources: []` в done-событии никак не отображается в UI как предупреждение.

Старый `build_prompt` при пустых чанках сохранял строгий RAG-промпт → модель явно говорила «не нашёл в документах».

**Сценарий:** Пользователь думает, что ответ основан на его документах, а он взят из параметрики.

**Исправление — два варианта:**
- (A) При `context_paths` задан, но `chunks == []` — использовать строгий RAG-промпт (старое поведение): модель скажет «не нашёл».
- (B) Оставить общий промпт, но добавить в `done`-событие флаг `no_context_found: true` и показывать в UI предупреждение «Документы не содержат ответа на этот вопрос».

Вариант B удобнее для пользователя. Вариант A безопаснее по умолчанию.

### 5. `<think>` тег виден в блоке рассуждений при leading-символе
**Файл:** `src/backend/memo/services/rag.py:165`

Открывающий `<think>` убирается только если `before.startswith("<think>")`. Если модель шлёт `\n<think>рассуждение</think>ответ`, то `before = "\n<think>рассуждение"` → `startswith` возвращает `False` → тег попадает в UI.

**Исправление:**

```python
# вместо startswith — strip + lstrip тега
before = before.strip()
if before.startswith("<think>"):
    before = before[len("<think>"):]
```

## Средние (неочевидное поведение, редкий сценарий)

### 6. `_auto_index` блокирует отправку HTTP-заголовков
**Файл:** `src/backend/memo/api/chats.py:104`

`await _auto_index()` выполняется полностью до того, как `StreamingResponse` сконструирован и заголовки отправлены клиенту. При большом устаревшем контексте пользователь ждёт несколько минут с пустым экраном.

**Исправление:** Сделать `_auto_index` фоновой задачей через `asyncio.create_task`, или пробрасывать прогресс переиндексации как SSE-события в начале стрима.

### 7. Несоответствие system prompt через историю
**Файл:** `src/backend/memo/services/rag.py:82`

System prompt выбирается на основе текущего turn'а (нашли ли чанки сейчас), но history содержит ответы, сгенерированные под другим промптом. Если Turn 1 был RAG, а Turn 2 — без чанков, модель получает `_GENERAL_SYSTEM_PROMPT` но в контексте видит RAG-ответ из прошлого витка. Может провоцировать выдумывание продолжения цитат.

**Исправление:** Хранить в сессии (или в метаданных чата) стабильный system prompt, не менять его между турнами одного чата.

---

### 8. Пустой `content` в истории создаёт два подряд user-turn
**Файл:** `src/backend/memo/services/rag.py:86`

`_build_messages` пропускает history-записи где `content` — пустая строка (прерванный ответ). Если прерванный ассистентский turn был сохранён с `content=''`, он выпадает из истории → в `messages` оказывается два подряд `{role: "user"}`. Некоторые модели возвращают 400 на такой последовательности.

**Исправление:** Включать записи даже с пустым content (с плейсхолдером `"[нет ответа]"`), или фильтровать пары целиком (и user, и следующий assistant).
