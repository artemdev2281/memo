## Высокие (неверное поведение, заметное пользователю)

### 1. Снимок (mtime,size) рассогласован с file_hash → правки во время индексации теряются навсегда
**Файл:** `src/backend/memo/services/indexer.py:60` (+ `:150`, `:203`)

`file_hash` вычисляется в `load(path)` (строка 150, снимок A), затем идёт
многосекундный `ollama.embed` (строка 168), и только потом `_db_upsert` зовёт
`_stat` (строка 60), записывая mtime/size **текущего** файла (снимок B). Если
пользователь сохранил новое содержимое в этом окне, базовая линия становится
`(hash_A, mtime_B, size_B)` — несогласованной.

**Сценарий:** правка файла попадает в окно между `load` и `_db_upsert`. Позже
`mark_changed_stale` делает `os.stat`, видит `mtime/size == сохранённым (B)` →
срабатывает дешёвый предфильтр → файл **не перехэшируется и не помечается
stale**. Индекс навсегда отдаёт содержимое A, хотя на диске — B. Это тихая
потеря корректности в самом механизме устаревания, который этот пасс усиливал.

**Исправление:** снимать `os.stat` одновременно с хэшем (в `load()`/в начале
обработки файла в `index_files`) и пробрасывать `(mtime,size)` в `_db_upsert`,
а не читать их повторно после эмбеддинга.

### 2. stale_warning для статуса «error» → ложное предупреждение на каждом ответе в папке-контексте
**Файл:** `src/backend/memo/services/rag.py:136`

Условие расширено до `row.status in ("stale", "error")`. Неподдерживаемые файлы
(`.xlsx`, `.png` и т.п.) в папке-контексте получают статус `error` («Unsupported
format») и **больше не ретраятся** (расширение не в `SUPPORTED`), оставаясь в
этом статусе навсегда.

**Сценарий:** контекст — папка с `report.xlsx`. После первого сообщения у него
статус `error`. На каждом последующем ответе `answer_stream` обходит все
expanded-пути, натыкается на `error` и ставит `stale_warning=True`. Пользователь
видит предупреждение «контекст устарел/неполон» на **каждом** ответе, хотя все
реальные документы свежие. До этого дифа (только `stale`) предупреждение молчало
— это регрессия.

**Исправление:** не учитывать в предупреждении перманентно-неиндексируемые файлы
(неподдерживаемые форматы), либо предупреждать только по `stale`, а `error`
показывать отдельным каналом.

## Средние (неочевидное поведение / редкий сценарий)

### 3. Синхронное хэширование на event loop блокирует все запросы бэкенда
**Файл:** `src/backend/memo/api/chats.py:135`

`_files_needing_index()` → `mark_changed_stale()` синхронно читает и хэширует
(sha256) каждый `indexed`-файл контекста прямо внутри async-генератора `stream()`
— без `await`, без потока.

**Сценарий:** отправка сообщения в чат с большой папкой-контекстом. Хэширование
сотен МБ блокирует event loop на секунды, замораживая health-check, IPC выдачи
порта и любые параллельные запросы. Несогласованность: `reconcile_all` на старте
вынесен в daemon-поток, а это идентичное хэширование в горячем пути запроса — нет.

**Исправление:** вынести хэширование в поток (`asyncio.to_thread` / executor),
как сделано для `reconcile_all`.

### 4. answer_started по thinking-дельте отбрасывает накопленный content_buf
**Файл:** `src/backend/memo/services/rag.py:164`

При первой `thinking`-дельте ставится `answer_started = True`. Если до этого
момента уже пришла `content`-дельта, она осела в `content_buf` и теперь не
сбрасывается ни здесь, ни финальным fallback (он требует `not answer_started`).

**Сценарий:** reasoning-модель, отдающая `content`-дельту раньше первой
`thinking`-дельты → начало ответа теряется молча. Маловероятный порядок (обычно
thinking идёт первым; qwen3 на текущей Ollama вообще не использует поле
`thinking`), но при смене модели/версии реализуем.

**Исправление:** при переходе в `answer_started=True` сбрасывать накопленный
`content_buf` как `token`.

### 5. Исключение в индексации внутри потока обрывает SSE без события об ошибке
**Файл:** `src/backend/memo/api/chats.py:135`

Раньше `await _auto_index()` выполнялся **до** `return StreamingResponse`, поэтому
сбой давал чистый HTTP 500 с detail. Теперь индексация выполняется после отправки
заголовков, внутри `stream()`.

**Сценарий:** непредвиденное исключение в `_files_needing_index`/`mark_changed_stale`
(ошибка БД, не-`OSError` при хэшировании) поднимается после старта стрима →
клиент видит обрыв соединения без `{"type":"error"}`-события → фронтовый
`for await` молча завершается, пользователь не видит причины.

**Исправление:** обернуть индексацию внутри `stream()` в try/except и отдавать
`{"type":"error", ...}` при сбое.

## Примечание по латентности think=OFF (не дефект кода)
Эмпирически на Ollama 0.9.2 + qwen3:4b: модель **игнорирует `think=false`** и всё
равно генерирует полный CoT (~2000–2400 симв.) до `</think>`; `/no_think` в
user/system-сообщении его не убирает. Ответ при этом стримится корректно (132
токен-события), но первый токен появляется только после скрытой фазы рассуждений
(~25с). Это особенность модели/сборки Ollama, а не баг разделителя. Возможные
улучшения UX: индикатор работы при think=OFF, либо роутинг быстрых ответов на
не-reasoning модель (`mistral:latest`).
